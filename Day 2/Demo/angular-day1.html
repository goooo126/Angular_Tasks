🅰️ What is Angular?

Angular is a TypeScript-based front-end framework built by Google for creating Single Page Applications (SPAs). It provides everything out-of-the-box: routing, forms, HTTP, testing, and more.

Key Features

  • Component-based architecture
  • TypeScript by default
  • Two-way data binding
  • Dependency Injection built-in
  • Angular CLI for scaffolding
  • RxJS for reactive programming

Getting Started

# Install Angular CLI
npm install -g @angular/cli

# Create new project
ng new my-app

# Run development server
cd my-app
ng serve

# Open http://localhost:4200
Project Structure: src/app/ contains your components, services, and modules. angular.json configures the build. tsconfig.json configures TypeScript.

📁 Angular Project Structure

After running ng new my-app, Angular generates a complete project folder. Understanding where things live is essential before writing any code.

Folder Structure

my-app/
├── public/              # Static files (images, icons)
│   └── favicon.ico
├── src/
│   ├── app/             # Your components & logic
│   │   ├── app.component.ts
│   │   ├── app.component.html
│   │   ├── app.component.css
│   │   └── app.module.ts
│   ├── index.html         # Main HTML page
│   ├── main.ts            # App entry point
│   └── styles.css         # Global styles
├── angular.json         # Build & project config
├── package.json         # Dependencies
└── tsconfig.json        # TypeScript config

Key Files & Folders

PathPurpose
src/app/All your components, services, models live here
public/Static files served at root URL (images, fonts, etc.)
angular.jsonProject config — styles, scripts, build options
app.module.tsRoot module — registers all components
app.component.*Root component — the starting point of your app
styles.cssGlobal CSS — applies to the entire app
public/ folder: Any file you place in public/ is served at the root URL. So public/logo.png is accessed in your code as just 'logo.png' — no path prefix needed.
Why does this matter? When you generate components with the CLI, they go inside src/app/. When you add images, they go in public/. When you add Bootstrap, you configure it in angular.json. Knowing the structure saves you from guessing where things belong.

📊 Angular Versions – v1 vs v2+

AngularJS (v1) and Angular (v2+) are two completely different frameworks. Angular 2+ was a full rewrite — not an upgrade.
FeatureAngularJS (v1)Angular (v2+)
LanguageJavaScriptTypeScript
ArchitectureMVC (Model-View-Controller)Component-based
Data BindingTwo-way (scope-based)One-way by default, two-way opt-in
Mobile SupportNot built for mobileMobile-first design
CLINo official CLIPowerful Angular CLI
PerformanceSlower (digest cycle)Faster (change detection, AOT)
Dependency InjectionBasic DI by nameHierarchical DI by type
RenderingDOM-basedUses Renderer (platform-independent)
Why the rewrite? AngularJS had performance issues with large apps, lacked mobile support, and its architecture didn't scale well. Angular 2+ was redesigned from scratch with modern web standards.

Version Timeline

  • AngularJS (2010) – The original, JavaScript-based
  • Angular 2 (2016) – Complete rewrite in TypeScript
  • Angular 4-17+ (2017–now) – Incremental improvements, same core
Note: When people say "Angular" today, they mean v2+. "AngularJS" specifically refers to v1. AngularJS reached end of life in December 2021.

🎀 Decorators

A Decorator is a special function that attaches metadata to a class, method, or property. Angular uses decorators to tell the framework how to process and configure classes. They start with @.
Why? Without decorators, Angular wouldn't know that your class is a component, a service, a module, or a directive. Decorators are how you tell Angular what a class is and how it should behave.

Common Angular Decorators

DecoratorUsed For
@Component()Marks a class as a component
@NgModule()Marks a class as a module
@Injectable()Marks a class as a service (for DI)
@Directive()Marks a class as a directive
@Pipe()Marks a class as a pipe
@Input()Receives data from parent component
@Output()Sends data to parent component

@Component Example

import { Component } from '@angular/core';

@Component({
  selector:    'app-header',
  templateUrl: './header.component.html',
  styleUrls:   ['./header.component.css']
})
export class HeaderComponent {
  title = 'My App';
}

@Injectable Example

import { Injectable } from '@angular/core';

@Injectable({
  providedIn: 'root'
})
export class DataService {
  getData() { return ['A', 'B']; }
}
Key Point: Decorators are TypeScript features that Angular leverages heavily. The @ symbol followed by a function call is the decorator syntax. The object you pass inside is the metadata.

🛠️ Creating Components

The Angular CLI generates components automatically with the correct structure, files, and module registration. You rarely create component files manually.
Why CLI? It creates 4 files (.ts, .html, .css, .spec.ts), registers the component in the module, and follows Angular naming conventions — saving time and preventing mistakes.

CLI Commands

# Full command
ng generate component header

# Shorthand
ng g c header

# Creates:
# src/app/header/
#   ├── header.component.ts
#   ├── header.component.html
#   ├── header.component.css
#   └── header.component.spec.ts

# Useful flags
ng g c header --inline-style     # no CSS file
ng g c header --inline-template  # no HTML file
ng g c header --skip-tests       # no spec file
ng g c shared/card               # in subfolder

Generated Code

// header.component.ts
import { Component } from '@angular/core';

@Component({
  selector: 'app-header',
  templateUrl: './header.component.html',
  styleUrls: ['./header.component.css']
})
export class HeaderComponent {

}
<!-- Using it in another template -->
<app-header></app-header>

Component = 3 Parts

  • TypeScript Class (.ts) – Logic & data
  • HTML Template (.html) – The view
  • CSS Styles (.css) – Appearance

📄 Templates – Internal vs External

A template is the HTML view of a component. Angular supports two ways to define templates: external (separate file) or inline (inside the component class).

External Template (Recommended)

@Component({
  selector: 'app-user',
  templateUrl: './user.component.html',
  styleUrls: ['./user.component.css']
})
export class UserComponent {
  name = 'Mahmoud';
}
<!-- user.component.html -->
<div class="user-card">
  <h2>{{ name }}</h2>
  <button (click)="edit()">Edit</button>
</div>

Tip Use external templates for any component with more than 3-4 lines of HTML.

Inline Template

@Component({
  selector: 'app-badge',
  template: `
    <span class="badge">
      {{ label }}
    </span>
  `,
  styles: [`
    .badge {
      background: #dd0031;
      color: white;
      padding: 4px 12px;
      border-radius: 12px;
    }
  `]
})
export class BadgeComponent {
  label = 'New';
}

Note Use template + backticks for small, simple components.

Key Difference: templateUrl points to an external file. template contains inline HTML directly. Same applies to styleUrls vs styles.

🔤 Interpolation {{ }}

Interpolation is the simplest form of data binding. It lets you embed component properties and expressions into your HTML template using double curly braces {{ }}.
Why? Interpolation is how you display dynamic data from your component class in the view. Whenever the data changes, Angular automatically updates the DOM.

Template

<!-- Display a property -->
<h1>{{ title }}</h1>

<!-- Display with text -->
<p>Welcome, {{ username }}!</p>

<!-- Expressions -->
<p>Total: {{ price * quantity }}</p>

<!-- Method calls -->
<p>{{ getName().toUpperCase() }}</p>

<!-- Ternary operator -->
<p>{{ isActive ? 'Active' : 'Inactive' }}</p>

<!-- String concatenation -->
<p>{{ 'Hello, ' + username }}</p>

Component Class

@Component({
  selector: 'app-home',
  templateUrl: './home.component.html'
})
export class HomeComponent {
  title = 'My App';
  username = 'Mahmoud';
  price = 50;
  quantity = 3;
  isActive = true;

  getName(): string {
    return this.username;
  }
}
Rules:
  • You can use expressions, method calls, ternary operators
  • You cannot use assignments (=), new, chained expressions (;), or increment/decrement (++/--)

🔗 Property Binding [ ]

Property binding lets you set an HTML element's DOM property to a component property value using square brackets [ ]. It flows data one-way: Component → View.
Why not interpolation? Interpolation only works for text content. For non-string properties like disabled, hidden, src, or colspan — you need property binding.

Property Binding Examples

<!-- Bind to DOM properties -->
<img [src]="imageUrl">
<button [disabled]="!isValid">Submit</button>
<p [textContent]="message"></p>
<div [innerHTML]="htmlContent"></div>

<!-- Attribute Binding -->
<td [attr.colspan]="colSpan"></td>
<div [attr.aria-label]="label"></div>

<!-- Class Binding -->
<div [class.active]="isActive"></div>
<div [class.text-danger]="hasError"></div>

<!-- Style Binding -->
<div [style.color]="textColor"></div>
<div [style.font-size.px]="fontSize"></div>

Component Class

export class DemoComponent {
  imageUrl = 'assets/logo.png';
  isValid = false;
  message = 'Hello World';
  colSpan = 2;
  isActive = true;
  hasError = false;
  textColor = 'red';
  fontSize = 18;
}

Interpolation vs Property Binding:

<!-- These are equivalent for strings: -->
<p>{{ message }}</p>
<p [textContent]="message"></p>

<!-- But for non-strings, use binding: -->
<button [disabled]="isDisabled">
  <!-- ✅ Correct -->
</button>
<button disabled="{{ isDisabled }}">
  <!-- ❌ Won't work properly -->
</button>

🔁 *ngFor – Loop Rendering

*ngFor is a structural directive that repeats a template for each item in a list. It's Angular's way of rendering arrays and iterables in the DOM.
Why? Most apps need to display lists — users, products, messages, etc. *ngFor iterates over an array and creates DOM elements for each item automatically.

Basic Usage

<!-- Basic loop -->
<ul>
  <li *ngFor="let item of items">
    {{ item.name }}
  </li>
</ul>

<!-- With index and helpers -->
<div *ngFor="let item of items;
            let i = index;
            let first = first;
            let last = last;
            let even = even">
  <span>{{ i + 1 }}. {{ item.name }}</span>
  <span *ngIf="first">(First!)</span>
</div>

<!-- trackBy for performance -->
<li *ngFor="let item of items;
          trackBy: trackById">
  {{ item.name }}
</li>

Component Class

export class ProductComponent {
  items = [
    { id: 1, name: 'Phone', price: 999 },
    { id: 2, name: 'Laptop', price: 1999 },
    { id: 3, name: 'Tablet', price: 599 }
  ];

  trackById(index: number, item: any): number {
    return item.id;
  }
}

Available Local Variables

VariableTypeDescription
indexnumberIndex of the current item
firstbooleanTrue if first item
lastbooleanTrue if last item
evenbooleanTrue if even index
oddbooleanTrue if odd index
Best Practice: Always use trackBy when rendering large lists. It helps Angular identify which items changed, improving performance by avoiding unnecessary DOM re-renders.
