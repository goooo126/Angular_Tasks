⚙️ How TypeScript Works

TypeScript is a superset of JavaScript that adds static type checking. Every valid JS is valid TS – TypeScript just adds types on top.
Why? Types catch bugs at compile time, not at runtime. Most modern frameworks require TypeScript.

The Compilation Flow

// You write TypeScript (.ts)
let name: string = "TypeScript";

        ⬇️  tsc compiles

// Browser reads JavaScript (.js)
let name = "TypeScript";

TypeScript never runs in the browser. It always compiles to plain JavaScript first.

2 / 27

🔄 Compilation Process (TS → JS)

TypeScript uses the TypeScript Compiler (tsc) to transform .ts files into .js files. Type annotations are stripped away — they exist only at development time.

1. Write

You write .ts files with types, interfaces, and modern syntax

2. Compile

tsc checks types and transforms code to the target JS version

3. Run

Browser/Node executes plain .js — no types remain at runtime

# Install TypeScript globally
npm install -g typescript

# Compile a single file
tsc app.ts              # → produces app.js

# Watch mode (auto-recompile on save)
tsc --watch

📦 Variables

Variables are named containers that store values. TypeScript adds type annotations so the compiler knows what type of data each variable holds.
Why? Real-world applications use typed variables everywhere. Type safety prevents passing wrong data between parts of your code.

Syntax

let   variableName: type = value;    // can be reassigned
const variableName: type = value;    // cannot be reassigned
var   variableName: type = value;    // avoid (function-scoped, old style)

Examples

let username: string = "Mahmoud";
let age: number = 25;
let isActive: boolean = true;
const API_URL: string = "/api/v1";

// Type inference – TS can figure it out
let city = "Cairo";  // inferred as string

🎨 Data Types – Practical Examples

Arrays & Tuples

// Array of one type
let scores: number[] = [90, 85, 78];
let names: Array<string> = ["Ali", "Sara"];

// Tuple – fixed structure
let user: [string, number] = ["Mahmoud", 25];

// Array of objects

  { name: "Phone", price: 999 },
  { name: "Laptop", price: 1999 }
];

Special Types

// any – avoid when possible
let data: any = "hello";
data = 42;         // no error, but risky

// unknown – safe alternative
let input: unknown = "hello";
// input.toUpperCase(); // ❌ Error!
if (typeof input === "string") {
  input.toUpperCase();  // ✅ OK after check
}

// void – no return
function log(msg: string): void {
  console.log(msg);
}

🔧 Functions

Functions in TypeScript can have typed parameters and typed return values. This ensures you always call them with the right data and get the expected result back.
Why? Methods in classes, services, and utility modules are all typed functions. Type-safe functions prevent runtime errors.

Syntax

// Named function
function greet(name: string): string {
  return `Hello, ${name}!`;
}

// Arrow function
const add = (a: number, b: number): number => a + b;

// Void return type
function logMessage(msg: string): void {
  console.log(msg);
}

❓ Optional & Default Parameters

Optional parameters use ? — they may or may not be provided. Default parameters use = — they have a fallback value if omitted.
Why? Functions and class methods frequently have optional configuration parameters. This is a very common pattern in real-world code.

Optional Parameters

function greet(
  name: string,
  greeting?: string  // optional
): string {
  return greeting
    ? `${greeting}, ${name}!`
    : `Hello, ${name}!`;
}

greet("Ali");            // "Hello, Ali!"
greet("Ali", "Ahlan");  // "Ahlan, Ali!"

Default Parameters

function createUser(
  name: string,
  role: string = "viewer"  // default
): void {
  console.log(`${name} is ${role}`);
}

createUser("Sara");          // "Sara is viewer"
createUser("Sara", "admin"); // "Sara is admin"

🏗️ Classes

A class is a blueprint for creating objects. It bundles data (properties) and behavior (methods) together. TypeScript classes are like JavaScript classes but with types and access modifiers.
Why? Modern frameworks are built on classes. Classes are the foundation for organizing code into reusable, structured components.

Syntax & Example

class Person {
  name: string;
  age: number;

  constructor(name: string, age: number) {
    this.name = name;
    this.age = age;
  }

  introduce(): string {
    return `I'm ${this.name}, ${this.age} years old`;
  }
}

const p = new Person("Mahmoud", 25);
console.log(p.introduce());

🧬 Inheritance

Inheritance lets a class extend another class, inheriting its properties and methods. The child class can add new features or override existing behavior.
Why? Inheritance is used for shared logic — base classes, shared services, and abstract patterns.

Syntax & Example

class Animal {
  constructor(public name: string) {}

  speak(): string {
    return `${this.name} makes a sound`;
  }
}

class Dog extends Animal {
  speak(): string {
    return `${this.name} barks 🐕`;
  }
}

const dog = new Dog("Rex");
dog.speak();  // "Rex barks 🐕"

🔑 Access Modifiers (Keywords)

Access modifiers control who can access class members. TypeScript has three: public, private, and protected.
ModifierClassSubclassOutside
public✅✅✅
protected✅✅❌
private✅❌❌

  public    owner: string;
  private   balance: number;
  protected bankName: string;

  constructor(owner: string, bal: number) {
    this.owner = owner;
    this.balance = bal;
    this.bankName = "MyBank";
  }

  public getBalance(): number {

  }

🔒 Readonly

The readonly keyword marks a property as immutable after initialization. It can only be set in the declaration or constructor.
Why? Immutability is important in modern applications — especially for configuration and state management. Prevents accidental data mutation.

Syntax & Example

class Config {
  readonly apiUrl: string;
  readonly version: number = 1;

  constructor(url: string) {
    this.apiUrl = url;  // ✅ OK in constructor
  }
}

const cfg = new Config("https://api.com");
// cfg.apiUrl = "x"; // ❌ Error: readonly

📐 Interfaces

An interface defines the shape (structure) of an object — what properties and methods it must have. It's a contract that objects must follow.
Why? Interfaces are used everywhere — for data models, API responses, configuration objects, and function contracts.

Syntax & Example

interface User {
  id: number;
  name: string;
  email: string;
  avatar?: string;  // optional
}

function printUser(user: User): void {
  console.log(user.name);
}

const u: User = {
  id: 1,
  name: "Mahmoud",
  email: "m@test.com"
};

printUser(u); // ✅ OK

🏷️ Type Aliases

A type alias creates a custom name for any type — primitives, unions, tuples, objects, or complex combinations. Uses the type keyword.
Why? Reusable type definitions keep your code DRY. Great for union types, callback signatures, and API response shapes.

Syntax & Example

// Simple alias
type ID = string | number;

// Object type alias
type Product = {
  id: ID;
  name: string;
  price: number;
};

// Function type alias
type Callback = (data: string) => void;

// Tuple alias
type Coordinate = [number, number];

Interface vs Type Alias

FeatureInterfaceType
Objects✅✅
Extend✅ extends✅ & (intersection)
Unions❌✅
Primitives❌✅
Declaration merging✅❌

Tip Use interface for object shapes. Use type for unions, tuples, and primitives.

🎯 Enums

Enums define a set of named constants. They make code more readable by replacing magic numbers or strings with descriptive names.
Why? Applications use enums for user roles, status codes, and component states. They make switch statements and conditionals clearer.

Numeric & String Enums

// Numeric enum (auto-increments from 0)
enum Direction {
  Up,       // 0
  Down,     // 1
  Left,     // 2
  Right     // 3
}

// String enum (explicit values)
enum Role {
  Admin  = "ADMIN",
  Editor = "EDITOR",
  Viewer = "VIEWER"
}

let myRole: Role = Role.Admin;
console.log(myRole); // "ADMIN"

🔀 Union Types

A union type allows a value to be one of several types, separated by the pipe | operator. The variable can hold any of the listed types.
Why? Applications often deal with values that can be multiple types — loading states, API responses that might be data or an error, nullable values.

Syntax & Example

// Basic union
let id: string | number;
id = "abc";  // ✅
id = 123;    // ✅
// id = true; // ❌ Error

// Literal union (set of exact values)
type Status = "loading" | "success" | "error";

function setStatus(s: Status): void {
  console.log(s);
}

setStatus("loading");  // ✅
// setStatus("done");  // ❌ Error

➕ Intersection Types

An intersection type combines multiple types into one using the & operator. The resulting type has all properties of each combined type.
Why? Composing types is a core pattern — combining entity types with metadata, merging interface contracts, and extending API response shapes.

Syntax & Example

type HasName = { name: string };
type HasAge  = { age: number };

// Intersection: must have BOTH
type Person = HasName & HasAge;

const p: Person = {
  name: "Ali",
  age: 30
}; // ✅ Must have both properties

🔍 Type Narrowing

Type narrowing is when TypeScript refines a broad type to a more specific one based on runtime checks. After a check, TS "knows" the precise type.
Why? You often deal with values that could be multiple types (API responses, form values, function parameters). Narrowing prevents runtime errors.

Common Narrowing Techniques

// 1. typeof guard
function format(val: string | number): string {
  if (typeof val === "string") {
    return val.toUpperCase();   // TS knows: string
  }
  return val.toFixed(2);       // TS knows: number
}

// 2. Truthiness check
function greet(name?: string): string {
  if (name) {
    return `Hi, ${name}`;    // TS knows: string
  }
  return "Hi, guest";
}

// 3. instanceof guard
if (err instanceof Error) {
  console.log(err.message); // TS knows: Error
}

🧩 Generics

Generics let you write reusable code that works with any type while still maintaining type safety. Think of <T> as a type variable — a placeholder filled in when the code is used.
Why? Many libraries use generics: Array<T>, Promise<T>, Map<K, V>, and API response wrappers. Generics are essential for writing reusable, type-safe code.

Syntax & Example

// Generic function
function identity<T>(value: T): T {
  return value;
}

identity<string>("hello");  // T = string
identity<number>(42);       // T = number
identity(true);              // T inferred: boolean

// Generic interface
interface ApiResponse<T> {
  data: T;
  status: number;
  message: string;
}

const res: ApiResponse<User[]> = {
  data: [{ id: 1, name: "Ali" }],
  status: 200,
  message: "OK"
};

🧩 Generics – Constraints & Classes

Generic Constraints

// T must have a "length" property
function logLength<T extends { length: number }>(
  item: T
): void {
  console.log(item.length);
}

logLength("hello");    // ✅ string has length
logLength([1, 2]);    // ✅ array has length
// logLength(42);      // ❌ number has no length

Generic Class

class DataStore<T> {
  private items: T[] = [];

  add(item: T): void {
    this.items.push(item);
  }

  getAll(): T[] {
    return [...this.items];
  }
}

const users = new DataStore<User>();
users.add({ id: 1, name: "Ali" });

📦 Modules & ES Imports/Exports

TypeScript uses ES modules to organize code into separate files. Each file is a module. You export what others need and import what you need.
Why? Every TypeScript file uses imports/exports. Modules organize code into separate files and your app is a tree of module dependencies.

Syntax

// ── user.model.ts ──
export interface User {
  id: number;
  name: string;
}

export const DEFAULT_ROLE = "viewer";

// ── user.service.ts ──
import { User, DEFAULT_ROLE } from './user.model';

// Default export
export default class UserService { }

// Import default
import UserService from './user.service';

⚡ Constructor Shorthand

TypeScript allows you to declare and initialize class properties directly in the constructor parameters by adding an access modifier. This saves repetitive code.
Why? This is a very common pattern in TypeScript. Constructor shorthand saves repetitive code and is used extensively in real-world projects.

Without Shorthand (Verbose)

class UserComponent {
  private service: UserService;
  private router: Router;

  constructor(
    service: UserService,
    router: Router
  ) {
    this.service = service;
    this.router = router;
  }
}

With Shorthand (Clean Way)

class UserComponent {
  // One line does it all!
  constructor(
    private service: UserService,
    private router: Router
  ) {}
  // Properties are auto-created
  // and assigned from parameters
}

// ✅ this.service is available
// ✅ this.router is available

🛠️ Type Assertions & Utility Types

Type assertions tell TypeScript "trust me, I know the type." Utility types are built-in type transformers that modify existing types.

Type Assertions

// "as" syntax (preferred)
const input = document
  .getElementById("name") as HTMLInputElement;
input.value = "Mahmoud";

// Non-null assertion (!)
const el = document.querySelector(".box")!;
// Tells TS: "I guarantee this exists"

Common Utility Types

Partial<User>     // All props optional
Required<User>    // All props required
Readonly<User>    // All props readonly
Pick<User, 'name' | 'email'>  // Subset
Omit<User, 'id'>  // Exclude props
Record<string, number>  // Key-value map
